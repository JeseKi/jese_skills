# Guided Engineering Roadmap

## 用途

当用户想推进一个复杂工程目标，但不希望 Agent 过早直接实现代码时，使用这个 skill。

这个 skill 的目标是把一个大而模糊的目标，拆成有依赖、有产出、有验收标准、有项目事实依据的阶段化工程路线。

它是一套协作协议，不是某个技术栈的实现指南。

## 核心原则

- 先路线图，后实现。
- 只有进入某个阶段时，才展开该阶段细节。
- 如果阶段目标是理解当前系统，则由 Agent 负责基线调查。
- 用户默认负责学习、决策和实现；除非用户明确授权，否则 Agent 不直接代写完整功能代码。
- 每个阶段都必须说明它如何承接前一阶段，以及如何解锁后一阶段。
- 优先写项目事实，不写泛泛的学习建议。
- 使用用户的主要语言来写标题和说明。
- 必要技术术语保留原文，例如 ROS2、Nav2、TF、topic、action、API、database、compiler、runtime。

## 默认工作流

1. 如果目标不清楚，先简短澄清最终目标。
2. 创建或更新顶层路线图。
3. 顶层路线图只写阶段推进，不提前写完整实现细节。
4. 只为当前阶段创建目录，除非用户明确要求一次性创建所有阶段骨架。
5. 根据阶段类型补充当前阶段文档。
6. 如果当前阶段是基线阶段，Agent 应该检查项目并写出调查报告。
7. 如果当前阶段是实现引导阶段，Agent 应该先写接口契约、文件计划、验收条件和伪代码。
8. 除非用户明确要求实现，否则不要直接修改功能代码。
9. 每个阶段结束时，记录它解锁了哪些下一阶段工作。

## 顶层路线图要求

顶层路线图应该像 roadmap，而不是 README。

应该包含：

- 目标结果。
- 当前阶段。
- 阶段状态表。
- 阶段依赖图。
- 每个阶段一个小节。
- 每个阶段的目的。
- 每个阶段的依赖。
- 每个阶段的产出。
- 每个阶段的完成条件。
- 每个阶段完成后解锁的后续阶段。
- 路线图状态值和更新规则。

应该避免：

- 过早写未来阶段的完整实现细节。
- 写成长篇教程。
- 写成泛泛学习清单。
- 把所有未来阶段都当作已经在推进。
- 把顶层 `index.md` 写成项目介绍页。

## 推荐目录结构

除非用户指定其他位置，否则优先使用项目内文档目录。

示例：

```text
docs/<目标-roadmap>/
  index.md
  00-<基线阶段>/
    roadmap.md
    acceptance.md
    file-plan.md
    system-inventory.md
    risk-register.md
    notes.md
  01-<下一阶段>/
    roadmap.md
    acceptance.md
    file-plan.md
    notes.md
```

只在进入某个阶段时创建该阶段详细目录。除非用户明确要求，不要一次性创建所有未来阶段的详细文件。

## 阶段文档要求

大多数阶段可以使用：

- `roadmap.md`：这个阶段做什么，如何承接前一阶段，如何解锁后一阶段。
- `acceptance.md`：满足什么条件才算完成。
- `file-plan.md`：预计读取、新增、修改哪些文件，以及哪些文件不应该动。
- `notes.md`：决策记录、未确认问题、调试记录、阶段复盘。

基线阶段还可以使用：

- `system-inventory.md`：记录 package、模块、启动入口、运行组件、接口、数据流和当前行为。
- `risk-register.md`：记录基线调查中发现的风险，以及它们对后续阶段的影响。
- 领域专用链路文档，例如 `flag-pose-flow.md`、`auth-flow.md`、`data-pipeline-flow.md`、`request-lifecycle.md`。

## 阶段类型规则

### 基线阶段

当阶段目标是理解当前系统时，使用基线阶段规则。

Agent 应该检查项目并输出调查结论。

应该做：

- 只读取理解当前结构和数据流所需的文件。
- 识别 package、模块、启动入口、服务、topic、API、schema 或运行边界。
- 追踪当前系统中将被替换或扩展的行为链路。
- 区分已经由源码确认的事实和需要运行时确认的事项。
- 写出能解锁下一阶段的结论。

不应该做：

- 不要把基线阶段写成用户作业。
- 除非用户明确要学习计划，否则不要大量写“用户应该学习什么”。
- 不要修改功能代码。
- 不要把未知的运行时行为写成已经确认。

基线阶段应该回答：

```text
当前系统已经有什么？
当前系统如何工作？
当前能力边界在哪里？
哪些内容应该复用、替换或隔离？
下一阶段需要从这里得到什么？
```

### 架构阶段

当阶段目标是设计目标结构时，使用架构阶段规则。

Agent 应该提供：

- 模块或节点边界。
- 接口契约。
- 数据流。
- 状态归属。
- 失败模式。
- 迁移策略。

除非用户明确要求，不要直接实现。

### 实现引导阶段

当用户准备自己实现某个边界清晰的功能时，使用实现引导阶段规则。

Agent 应该提供：

- 文件计划。
- 接口契约。
- 伪代码。
- 验收条件。
- 调试清单。
- 常见错误。

除非用户明确说“帮我实现”或类似表达，否则不要写完整功能代码。

### 代码审阅阶段

当用户已经写了代码，希望获得反馈时，使用代码审阅阶段规则。

Agent 应该像工程师一样 review：

- 先列发现的问题，按严重程度排序。
- 尽量包含文件和行号引用。
- 重点关注 bug、行为回归、集成风险、缺失测试和接口不清晰。
- 给出具体修改建议，但不要默认重写整套方案。

### 集成阶段

当多个阶段需要串成端到端流程时，使用集成阶段规则。

Agent 应该提供：

- 启动或运行顺序。
- 集成假设。
- 调试检查点。
- 可观测性点位。
- 已知失败模式。
- 演示验收条件。

## 语言规则

匹配用户的主要语言。

如果用户主要用中文交流，则标题和说明都应该用中文。

技术标识符保持原样，例如：

```text
ROS2, Nav2, TF, topic, action, service, API, package, launch, base_link, cmd_vel
```

避免在中文文档中使用不必要的英文标题。

优先使用：

```text
阶段 00：系统基线
阶段产出
完成条件
依赖阶段
后续阶段
```

而不是：

```text
Stage 00: System Baseline
Expected Outputs
Exit Criteria
Depends On
Next Stage
```

## 行为约束

- 不要过度展开未来阶段。
- 不要制造虚假的确定性。
- 把未知事项标记为“需要运行时确认”。
- 不要把项目调查写成用户作业。
- 在路线图阶段或基线阶段，不要修改功能代码，除非用户明确要求。
- 不要把复杂 roadmap 写成普通 README。
- 不要让 speculative 的未来实现细节淹没当前阶段。
- 除非用户明确希望并行推进，否则不要让多个阶段同时变成 `进行中`。

## 阶段状态值

中文路线图优先使用：

```text
已计划    尚未开始，但已进入近期计划
进行中    当前正在推进
受阻      等待依赖阶段完成
已完成    已达到完成条件
暂缓      暂时不做，但不阻塞主线
返工      已完成过，但需要重新调整
```

英文路线图可以使用：

```text
Planned
In Progress
Blocked
Done
Deferred
Rework
```

## 实用判断规则

当用户说“不要直接实现”或“我想自己学着写”时：

- 提供路线图、接口、伪代码和 review 点。
- 不要提供完整实现。
- 修改代码前先询问或等待明确授权。

当用户说“这个阶段由你来梳理项目”时：

- 把该阶段当作基线调查。
- 检查仓库。
- 输出具体报告。
- 不要写成学习清单。

当用户说“创建当前阶段目录”时：

- 创建该阶段目录。
- 只创建该阶段类型需要的文件。
- 不要创建详细未来阶段目录，除非用户要求。

当用户说“顶层 index 应该是 roadmap”时：

- 使用状态表、依赖关系、阶段产出、完成条件和阶段流转。
- 移除 README 式介绍。

## 顶层路线图最小模板

```markdown
# 路线图：<目标名称>

## 目标结果

<最终系统行为或工程目标。>

## 当前路线状态

当前阶段：`00-xxx`

| 阶段 | 状态 | 依赖 | 阶段产出 |
| --- | --- | --- | --- |
| 00-xxx | 已计划 | 无 | <产出> |
| 01-yyy | 受阻 | 00-xxx | <产出> |

## 路线依赖图

```text
00-xxx
  -> 01-yyy
    -> 02-zzz
```

## 阶段 00：<名称>

状态：`已计划`

阶段目的：

- <目的>

阶段产出：

- <产出>

完成条件：

- <条件>

后续阶段：

- `01-yyy`

## 路线图更新规则

- 只有当前阶段可以进入 `进行中`。
- 阶段完成后，再创建或补全下一阶段的详细 roadmap。
- 如果某阶段发现前置假设错误，应回到依赖阶段更新结论。
```

## 基线阶段最小模板

```markdown
# 阶段 00：系统基线调查报告

## 本阶段定位

本阶段由 Agent 负责完成项目梳理，用于承接当前系统，并为后续阶段提供工程入口。

## 调查方式

<说明基于源码静态分析，还是也包含运行时采样。>

## 核心结论

<列出当前系统能力、缺失能力和关键边界。>

## 当前主线数据流

```text
<当前系统数据流>
```

## 对后续阶段的直接影响

### 对阶段 01：<名称>

<影响和切入点。>

## 本阶段结论

<说明解锁哪些后续阶段，以及推荐下一步。>
```
